Visitors who land on the site from a tagged link (e.g. our Chamber of Commerce profile, social media bios) will see their promo code automatically filled in for them, with a confirmation that the deal applied. Historical lead records from before today may be missing these fields. Going forward, everything the visitor types will be captured correctly.
- Made the contact forms' website URL field much more forgiving. Visitors can now type their site the way they naturally would (e.g. `yourbusiness.com` or `app.yourbusiness.com/contact`), not just the strict `www.yourbusiness.com` format that was previously required. The server automatically prefixes `https://` where needed.
- Tightened the layout of both contact forms so the new promo code field and the "Start the Conversation" button sit on the same row on larger screens, and stack cleanly on mobile.

- **Silent data-loss bug (discovered via DEV-678)**: both contact form payloads were sending a mix of camelCase and snake_case, but both server Zod schemas expect camelCase across the board. Zod silently strips unknown keys, so `current_website` → `currentWebsite`, `brief_summary` → `briefSummary` (details form / `/api/leads`), and `phone_number` → `phoneNumber`, `other_detail` → `otherDetail` (review form / `/api/audit`) were all arriving at the server as `undefined` and being stored as NULL. Fixed by renaming the outgoing payload keys only — internal react-hook-form field names, Zod schema keys, `register()` calls, JSX ids, and aria attributes remain snake_case on the review form (internal-only, not worth the cascading rename). **Historical submissions are permanently missing these fields**; nothing to do about it besides flag it. Exposed when the promo code migration added a NOT NULL-adjacent column that forced the mismatch into a crash.
- DEV-678 (URL validation): loosened `websiteUrlSchema` in `lib/validation/url.ts` from `^(https?:\/\/)?www\.[a-z0-9-]+(\.[a-z0-9-]+)+(?:\/[\S]*)?$/i` (required `www.`) to `^(https?:\/\/)?([a-z0-9-]+\.)+[a-z]{2,}(\/[\S]*)?$/i` so bare domains, non-www subdomains, and paths all pass. `normalizeWebsiteUrl()` unchanged — still prepends `https://` when missing before submit. Both contact forms consume the same schema so this is a one-line fix that hits both. Server repo is being loosened to match in parallel (same regex).
- DEV-678: added optional promo code field to both contact forms with URL auto-population. New `lib/promo-codes.ts` is the hardcoded registry (seeded with `NJCC2026` for the NJ Chamber of Commerce 2026 deal) — type, `findPromoCode`/`isValidPromoCode` helpers, and `active`+`expiresAt` kill switches. New `lib/hooks/use-promo-from-url.ts` reads `?promo=` via `useSearchParams`, validates against the registry, and persists matches to `sessionStorage['pvs_promo']` so a code from a tagged landing URL survives navigation across the site before the user reaches the form. URL wins over storage; unknown/inactive/expired codes from the URL are silently ignored (field stays blank rather than autopopulating with stale data). Both `contact-details-form.tsx` and `contact-review-form.tsx` got a `promoCode: z.string().trim().max(32).optional()` schema field, the hook wired via `useEffect`+`setValue` (no validation/dirty), an `<Input>` placed just above submit, and a small inline confirmation (`{label} applied`) shown only when the typed/populated value matches a *known active* promo. Payload uses the existing snake_case convention (`promo_code`) and is omitted entirely when blank, mirroring the `interestedIn` pattern. `clearStoredPromoCode()` runs on successful submit so a refresh doesn't re-fill the field. Wrapped the form slot in `ContactPageClient` with `<Suspense fallback={null}>` because `useSearchParams` requires it on prerendered pages — confirmed via two `npm run build` runs that all three contact routes still emit as `○ (Static)`. **Server validation:** intentionally none — we trust whatever the form submits and honor codes manually. The hardcoded list is a UX concern only; unknown codes typed by hand still POST through and land in Supabase/Discord. The site frontend will only ship after the parallel server PR lands, since the field has nowhere to persist until then. **Out of scope:** Calendly `/contact/call` (no form to autofill), promo dashboard, server-side validation. **Tradeoff:** the promo registry being hardcoded means deploys are required to add codes — acceptable for v1, will get migrated to a dashboard later.
